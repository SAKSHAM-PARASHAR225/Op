Unit 1 — Deadlock

Definition · 4 Conditions · System Model · RAG · Deadlock Formula

1.1

What is Deadlock?

▼

Deadlock is a situation where a group of processes are all stuck waiting forever — each one waiting for a resource held by another process in the same group. No process can ever proceed on its own.

Official Definition
Deadlock = A situation where two or more processes are permanently blocked, each waiting for a resource held by another — so no process can proceed.
Real-life analogy: Two cars on a one-lane bridge facing each other. Neither moves forward, neither reverses. Both wait forever.
Deadlock is a permanent block. The OS does NOT recover automatically — it must intervene.
Deadlock
  • All involved processes are BLOCKED permanently
  • None can make any progress at all
  • Cannot resolve by itself
Starvation
  • One process keeps being skipped over
  • Other processes ARE running normally
  • May eventually resolve itself
1.2

4 Necessary Conditions (Coffman Conditions)

▼

ALL four conditions must be true at the SAME TIME for deadlock to occur. Remove even ONE and deadlock is impossible. Identified by E.G. Coffman in 1971.

  • 1
    Mutual Exclusion — At least one resource can only be used by ONE process at a time. It is non-shareable. Example: a printer, a file open for writing.
  • 2
    Hold and Wait — A process is currently HOLDING at least one resource AND WAITING to acquire more resources that are currently held by other processes.
  • 3
    No Preemption — Resources cannot be forcefully taken away from a process. A process releases its resources only voluntarily when it is done using them.
  • 4
    Circular Wait — A circular chain of processes exists: P1 waits for P2's resource, P2 waits for P3's resource, ..., Pn waits for P1's resource. Forms a closed loop.
M-H-N-C — Mutual exclusion, Hold & wait, No preemption, Circular wait. All four must hold simultaneously for deadlock.
Deadlock PREVENTION works by ensuring at least one of these conditions can NEVER be true in the system design.
1.3

System Model for Deadlock

▼

A system model formally describes how processes and resources interact. Used to analyze and detect deadlock.

Components
  • Processes — P1, P2, ..., Pn (entities that compete for resources)
  • Resource Types — R1, R2, ..., Rm (CPU, memory, printers, locks, etc.)
  • Instances — Each resource type Ri has Wi instances (e.g., 3 printers = 3 instances)

A process uses a resource in 3 steps:

1. Request

Process asks OS for the resource. If unavailable, process enters waiting state.

2. Use

Process actively uses the resource to perform its work.

3. Release

Process voluntarily frees the resource so others can use it.

1.4

Resource Allocation Graph (RAG)

▼
Definition

A directed graph that visually shows the current state of resource allocation. It helps detect deadlocks without running complex algorithms.

Graph Components (Nodes)
  • Circle (○) = Process node
  • Rectangle (□) = Resource type node
  • Dots inside rectangle = individual instances
Graph Components (Edges)
  • P → R = Request edge — process is WAITING for this resource
  • R → P = Assignment edge — resource is GIVEN to this process

Deadlock Detection Rules

  • 1
    No cycle in graph → Definitely NO deadlock. System is safe.
  • 2
    Cycle + only 1 instance per resource type → Definitely DEADLOCK.
  • 3
    Cycle + multiple instances per resource type → POSSIBLE deadlock (need further analysis).
Example: RAG with DEADLOCK

P1→R1 (P1 wants R1), R1→P2 (R1 given to P2), P2→R2 (P2 wants R2), R2→P1 (R2 given to P1). Cycle: P1→R1→P2→R2→P1. Single instance per resource. → DEADLOCK

Example: RAG with Cycle but NO DEADLOCK

R1 has 2 instances. P1 holds one, P3 holds one. P2 requests R1 and waits. Cycle exists, but P3 will finish and release R1, allowing P2 to proceed. → NO DEADLOCK

RAG with cycle + single instance = certain deadlock. RAG with cycle + multiple instances = possible deadlock only.
1.5

Important Formula — Min/Max Resources for Deadlock

▼

Given n processes where each process Pi needs a maximum of xi units of a resource. In worst case, each process holds (xi-1) units and waits for 1 more. One extra unit breaks the deadlock chain.

Deadlock Resource FormulaMax units that ENSURE Deadlock = (x₁ + x₂ + ... + xₙ) - n = Σxi - n Min units that ENSURE NO Deadlock = (Σxi - n) + 1 Where: n = number of processes, xi = maximum need of process Pi
Logic: If each process holds (xi-1) units, total held = Σxi - n. They're all waiting. Add 1 more unit → one process can complete → chain reaction → no deadlock.

Unit 2 — Deadlock Handling

Prevention (4 ways) · Avoidance · Detection & Recovery · Ignorance

2.1

Overview of 4 Deadlock Handling Strategies

▼
Prevention

Break one of the 4 Coffman conditions at design time. Most restrictive.

Avoidance

Check safe state before every allocation (Banker's Algorithm). Smart but expensive.

Detection & Recovery

Let deadlock happen, detect it, then recover. Reactive approach.

Ignorance

Pretend deadlock doesn't exist. Used by Windows & Linux (Ostrich Algorithm).

P-A-D-I = Prevention → Avoidance → Detection → Ignorance. From most proactive to most passive.
2.2

Deadlock Prevention — Breaking the 4 Conditions

▼
  • 1
    Violate Mutual Exclusion
    Make all resources shareable. Problem: Not possible — printers, tape drives, etc. are inherently non-shareable. This approach is impractical for real systems.
  • 2
    Violate Hold and Wait
    A process must request ALL resources it will ever need BEFORE it starts. If it cannot get everything, it holds nothing and waits.
    Drawbacks: Wasteful (resources locked up early). Hard to predict all needs. Can cause starvation for processes needing many resources.
  • 3
    Violate No Preemption
    Allow OS to forcefully take a resource from a waiting process. Preempted resources are returned to the pool. The victim process restarts later.
    Only practical for saveable resources (CPU, memory pages). NOT for printers or disk arms.
  • 4
    Violate Circular Wait (Most Practical!)
    Assign a unique number to every resource type. Processes can only request resources in STRICTLY INCREASING order of these numbers. Circular chains become impossible.
    Result: May cause starvation but NEVER causes deadlock.
Most practical approach = violate Circular Wait by enforcing ordering on resource requests. Simple to implement, no runtime overhead.
2.3

Deadlock Avoidance — Safe State Concept

▼

In avoidance, the OS checks before every allocation: "If I grant this request, will the system remain in a SAFE state?" If yes → grant. If no → make process wait.

Safe State

A system is in a SAFE STATE if there exists at least one safe sequence — an ordering of all processes where each process can obtain all its required resources, complete, and release them, allowing the next process to proceed. Safe state guarantees NO deadlock will ever occur.

Unsafe State

No safe sequence exists. Deadlock MAY occur depending on future requests (not certain, but possible). The OS avoids entering unsafe states proactively.

Safe → Deadlock impossible. Unsafe → Deadlock possible but not guaranteed.
Banker's Algorithm (Unit 3) is the primary example of deadlock avoidance. It requires each process to declare maximum resource needs upfront — often impractical in real systems.
2.4

Deadlock Detection

▼

The OS allows deadlocks to occur, then periodically detects them using algorithms.

Single Instance Resources

Look for cycles in the RAG (Resource Allocation Graph). A cycle in the RAG with single instances = deadlock with 100% certainty.

Multiple Instances per Resource

A cycle alone is not sufficient. Must run the Safety Algorithm — convert RAG to Allocation and Request matrices, then check for a safe sequence. No safe sequence = deadlock.

OS can run detection periodically (every N seconds) or when CPU utilization drops significantly — a sign that processes might be blocked in deadlock.
2.5

Deadlock Recovery

▼

Recovery via Resources

Resource Preemption

Take a resource away from one process (the "victim") and give it to the deadlocked process. The victim is rolled back and retries later. Must choose victim carefully to minimize wasted work.

Rollback to Safe State

OS saves periodic checkpoints. On detecting deadlock, rollback the entire system to the last saved safe checkpoint. All changes after that checkpoint are undone. Expensive but complete.

Recovery via Processes

Kill One Process

Terminate the process that has done the least work (minimum loss). This releases all its resources. The process must restart from the beginning.

Kill All Deadlocked Processes

Terminate all processes in the deadlock. Completely breaks the deadlock but wastes all work done by those processes. Most drastic option.

2.6

Deadlock Ignorance — Ostrich Algorithm

▼
Ostrich Algorithm

The OS simply ignores the possibility of deadlock, like an ostrich burying its head in the sand. If deadlock occurs, the user notices the system freeze and manually reboots or kills the stuck processes.

Both Windows and Linux use the Ostrich Algorithm. Reasoning: deadlocks are very rare in practice. The overhead of prevention/avoidance/detection costs more than occasionally dealing with a rare deadlock manually.
This is the MOST WIDELY USED method in real-world operating systems!

Unit 3 — Banker's Algorithm

Safe State · 4 Data Structures · 3-Step Request · Safety Algorithm · Limitations

3.1

What is Banker's Algorithm?

▼
Definition

Banker's Algorithm is a deadlock avoidance algorithm. Before every resource allocation, it simulates granting the request and checks whether the resulting state is safe. If safe → grant. If unsafe → deny and make process wait. Named after banking: a bank only lends money when confident it can still meet all customers' needs.

Developed by Edsger Dijkstra. Works for systems with multiple resource types, each with multiple instances. Every process must declare its MAXIMUM resource needs at startup.
3.2

Four Data Structures

▼

Assume n processes and m resource types:

Available [1 × m]

Available[j] = currently free instances of resource type j. Updates as resources are allocated and released.

Max [n × m]

Max[i][j] = maximum instances of resource j that process i may ever request. Declared when process starts. Cannot change.

Allocation [n × m]

Allocation[i][j] = instances of resource j currently allocated to process i right now.

Need [n × m] — Most Important!

Need[i][j] = Max[i][j] − Allocation[i][j]. How many more instances process i might still request before completing.

Most Important FormulaNeed[i][j] = Max[i][j] − Allocation[i][j] ALWAYS compute the Need matrix FIRST when solving problems!
3.3

Banker's Algorithm — 3-Step Request Process

▼

When process Pi makes a resource request Request[i]:

  • 1
    Validity Check: Is Request[i] ≤ Need[i]?
    YES → go to Step 2.   NO → Error! Process exceeded declared maximum. Abort request.
  • 2
    Availability Check: Is Request[i] ≤ Available?
    YES → go to Step 3.   NO → Not enough free resources. Process must WAIT.
  • 3
    Pretend-and-Test: Temporarily grant the request:
    • Available = Available − Request[i]
    • Allocation[i] = Allocation[i] + Request[i]
    • Need[i] = Need[i] − Request[i]
    Run Safety Algorithm. If SAFE → actually grant ✓. If UNSAFE → rollback all changes, process waits ✗.
3.4

Safety Algorithm — Finding the Safe Sequence

▼

Determines whether the current state is safe. Tries to find an order in which all processes can complete.

Steps
  1. Set Work = Available (working copy of free resources)
  2. Set Finish[i] = false for all i (no process finished yet)
  3. Find any process Pi where: Finish[i] = false AND Need[i] ≤ Work
  4. If found: Work = Work + Allocation[i], set Finish[i] = true, go to Step 3
  5. If all Finish[i] = true → SAFE STATE ✓. Else → UNSAFE STATE ✗
The order in which processes are selected forms the Safe Sequence. Multiple valid safe sequences may exist.
Work = available resources pool. Each time a process Pi finishes, its allocated resources return to Work. Next process that fits gets to run. Chain reaction to completion.
3.5

Limitations of Banker's Algorithm

▼
Limitations
  • Processes must declare maximum needs upfront — often unknown
  • Safety check overhead on EVERY allocation request
  • Only works when number of processes and resources is fixed
  • Cannot handle dynamic resource types added at runtime
  • Does not work if a process can claim unlimited resources
Key Facts (Q&A)
  • Safe state = guaranteed no deadlock
  • Unsafe state = possible deadlock (not certain)
  • A system can be safe with only 1 free resource if a safe sequence exists
  • Banker's cannot prevent ALL deadlock situations (dynamic demands)

Unit 4 — Memory Management

Address Binding · Dynamic Loading · Dynamic Linking · Logical vs Physical Address · Base/Limit Registers

4.1

Memory Management — Introduction

▼
Definition

Memory Management is the process of controlling and coordinating computer memory — assigning portions (blocks) to various running programs to optimize overall system performance. Main memory (RAM) is where programs must reside to execute.

Goals: Efficient memory use · Protection between processes · Support multiprogramming · Fast access for CPU
4.2

Address Binding — 3 Types

▼
Definition

Address Binding is mapping the program's logical (virtual) addresses to corresponding physical (RAM) addresses. Must happen before the CPU can execute the program.

  • 1
    Compile-Time Binding (Static) — Compiler directly generates absolute physical addresses. Program must always load at the same fixed location. Oldest method. If location occupied → program cannot run.
  • 2
    Load-Time Binding — Compiler generates relocatable code (relative to address 0). The OS loader converts to absolute addresses when loading. If program moves → must reload entirely.
  • 3
    Execution-Time Binding (Dynamic) — Address translation happens at runtime by the MMU hardware. Process can be moved during execution. Used by ALL modern operating systems.
Compile-time = fixed. Load-time = set at loading. Execution-time = translated on-the-fly. Most flexible = Execution-time.
4.3

Dynamic Loading

▼
Definition

Dynamic Loading = a routine is loaded into RAM ONLY when it is called during execution. If never called, it is NEVER loaded. Also called Loading on Demand.

Advantages
  • Better memory utilization — unused routines never occupy RAM
  • Useful for large error handling or rarely-used code
  • No special OS support needed — done in program design
Note
  • The program itself manages which routines load when
  • OS does NOT automatically implement this
  • Programmer must design the program with dynamic loading in mind
4.4

Dynamic Linking

▼
Definition

Dynamic Linking postpones linking of library code until execution time. A small stub code locates and links the actual library routine at runtime. OS support IS required (unlike dynamic loading).

Static Linking
  • Library code copied INTO the executable at compile time
  • Every program has its own copy of library in its file
  • Larger executable files
  • No OS support needed
Dynamic Linking
  • Library stays separate (.dll in Windows, .so in Linux)
  • Multiple programs share ONE copy in RAM
  • Smaller executable files
  • OS support required to find and link at runtime
Linking is postponed until execution time. Dynamic linking enables shared libraries. Windows DLL = Dynamic Link Library.
4.5

Process Address Space — Logical vs Physical

▼
Logical Address Space

Set of ALL logical (virtual) addresses generated by a running process. CPU sees only logical addresses. For 32-bit system: 0 to 2^32 − 1. Process thinks it owns this entire space.

Physical Address Space

Set of actual physical RAM addresses. What the memory controller uses to access hardware. Much smaller than virtual address space typically.

Compile-time and load-time binding: logical = physical (same address). Execution-time binding: logical ≠ physical. MMU hardware does the translation every time.
Address Size CalculationPhysical address space = N words → Physical address size = log₂(N) bits Logical address space = L words → Logical address size = log₂(L) bits Steps: Convert bytes to words (divide by word_size), then take log₂
4.6

Base and Limit Registers — Hardware Protection

▼
Base Register

Holds the STARTING physical address of the current process. Set by the OS when a process is scheduled.

Limit Register

Holds the SIZE (length) of the current process's memory region. Defines the boundary.

CPU checks EVERY memory access: if Base ≤ Address ≤ Base + Limit → valid access. If outside → TRAP generated → OS terminates the process (segmentation fault). This prevents one process from corrupting another's memory.
This is called Hardware Address Protection. The check happens in CPU hardware automatically on every memory access — completely transparent to the program.
4.7

Key Components of Virtual Address Space

▼
Pages

Virtual memory divided into fixed-size blocks = pages. Basic unit of memory management.

Code Segment

Holds the executable instructions of the process — the program's machine code.

Data Segment

Global variables, heap (dynamically allocated memory). Holds program data during execution.

Page Table

OS-maintained structure mapping virtual page numbers to physical frame numbers.

Page Fault

When a process accesses a page not in RAM — OS must fetch it from disk.

Stack Segment

Local variables, function call stack, return addresses. Grows/shrinks dynamically.

Unit 5 — Memory Allocation Techniques

Swapping · Single Contiguous · Fixed Partition · Dynamic Partition · First/Best/Worst Fit · Compaction

5.1

Swapping

▼
Definition

Swapping = temporarily moving a process from RAM to secondary storage (disk) to free up memory. SWAP OUT: process moved to disk. SWAP IN: process brought back to RAM. Allows more processes than RAM can hold simultaneously.

Advantages
  • Allows more processes than RAM can hold
  • Increases degree of multiprogramming
  • Large processes can be accommodated
Disadvantages
  • Slow — disk I/O is much slower than RAM
  • Context switch time increases significantly
  • Performance degrades with heavy swapping
5.2

Single Contiguous Memory Allocation

▼

Simplest scheme. Memory split into exactly 2 parts: OS area (fixed, lower addresses) + one user process area (entire remaining RAM). Only ONE user process runs at a time. No multitasking. Used by earliest computer systems only.

5.3

Fixed Partition Scheme (Static Partitioning)

▼
Definition

Memory is divided into a fixed number of partitions at system startup. Sizes may be equal or unequal, but CANNOT change during operation.

Internal Fragmentation

Wasted space INSIDE an allocated partition. If 5KB process goes in 8KB partition → 3KB wasted inside. Cannot be used by anyone else. This is the MAIN PROBLEM of fixed partitions.

Advantages
  • Simple to implement and manage
  • Fast allocation — OS picks a partition quickly
  • Predictable behavior
Disadvantages
  • Internal fragmentation wastes memory
  • Fixed number of processes limits multitasking
  • Process larger than biggest partition cannot run
5.4

Dynamic (Variable) Partition Scheme

▼
Definition

When a process arrives, a partition is created that is EXACTLY the process size. No internal fragmentation. But after many allocate/deallocate cycles, free space becomes scattered.

External Fragmentation

Free memory is scattered in small non-contiguous blocks. Total free memory might be enough, but no single block is large enough for a new process. Solution = Compaction.

Advantages
  • No internal fragmentation
  • Efficient memory use
  • More processes can load simultaneously
Disadvantages
  • External fragmentation builds up over time
  • Need allocation algorithms to choose which hole to use
  • More complex to manage than fixed partitions
5.5

Dynamic Storage Allocation — First Fit, Best Fit, Worst Fit

▼
First Fit

Scan from start, use the FIRST hole large enough. Stop immediately when found. Fast. May fragment the beginning of memory.

Best Fit

Search entire list, use the SMALLEST hole that still fits. Minimizes wasted space per allocation. Leaves tiny fragments. Slower.

Worst Fit

Search entire list, use the LARGEST hole. Leaves biggest leftover (theoretically more useful). Generally the least efficient in practice.

First Fit and Best Fit are generally better than Worst Fit. First Fit = fastest. Best Fit = least waste per allocation. Worst Fit = generally avoided.
5.6

Compaction

▼
Definition

Rearranging all running processes in memory so all free blocks merge into ONE large contiguous block. Solves external fragmentation. REQUIRES execution-time (dynamic) address binding. Very expensive — every process must physically move and have its addresses updated.

How It Works
  • OS moves all processes to one end of RAM
  • All free space merges at the other end
  • Base registers of all processes are updated
Cost & Requirements
  • Requires dynamic address binding (execution-time)
  • Expensive — physically moves every process
  • System may pause during compaction

Unit 6 — Paging & Segmentation

Paging · TLB · Shared Pages · Multi-Level · Hashed · Inverted · Segmentation · Combined

6.1

Paging — Core Concept

▼
Definition

Paging = non-contiguous memory allocation scheme. Physical memory split into fixed-size FRAMES. Logical memory split into same-size PAGES. Any page can go into any available frame. A PAGE TABLE maps page number → frame number.

Advantages
  • Eliminates external fragmentation completely
  • Any page fits any frame — non-contiguous OK
  • Easy swapping (all same size)
  • Supports virtual memory efficiently
Disadvantages
  • Internal fragmentation (last page may not be full)
  • Page table requires extra memory
  • Two memory accesses per data access (without TLB)
Key Paging FormulasPage Number = Logical Address ÷ Page Size (integer division) Page Offset = Logical Address mod Page Size Physical Address = Frame Number × Page Size + Offset If logical space = 2^m, page size = 2^n: Page number = high-order (m−n) bits Offset = low-order n bits
6.2

TLB — Translation Lookaside Buffer

▼
Problem

Page table is in main memory → every memory access needs 2 accesses (1 for page table, 1 for data). Doubles access time!

Solution — TLB

A small, fast hardware cache storing recently used page table entries. Part of the CPU's MMU (Memory Management Unit).

TLB Hit

Page entry found in TLB → get physical address immediately → only ONE memory access needed. Fast path!

TLB Miss

Not in TLB → must access main memory page table → TWO accesses needed. The entry is then loaded into TLB.

Effective Access Time (EAT)EAT = (Hit Rate × Access_With_TLB_Hit) + (Miss Rate × Access_With_TLB_Miss) With TLB hit: 1 TLB lookup + 1 memory access With TLB miss: 1 TLB miss + 2 memory accesses (page table + data)
6.3

Shared Pages

▼
Concept

Multiple processes can have page table entries pointing to the SAME physical frame. Different logical pages map to the same physical page. Saves RAM significantly.

Requirements
  • Must be REENTRANT (code does not modify itself)
  • Must be READ-ONLY / write-protected
  • No writable data mixed with shared code
Use Cases
  • System libraries shared by many processes
  • Multiple users running same program (one code copy)
  • Inter-process communication via shared memory
6.4

Problem with Large Page Tables + Solutions

▼
The Problem

32-bit address space + 4KB pages → 2^20 entries × 4 bytes = 4MB per process for page table alone. With 100 processes → 400MB just for page tables. 64-bit space makes it far worse.

Solution 1: Multi-Level Paging

Break page table into levels. Level-1 → Level-2 → frames. Only pages actually in use need page table entries in memory. PTBR holds Level-1 table address.

Solution 2: Hashed Page Tables

Hash the virtual page number → hash table → linked list of entries. Good for 64-bit sparse address spaces. Fast lookup with hashing.

Solution 3: Inverted Page Tables

ONE entry per physical FRAME (not per logical page). Entry = (pid, page-number). Saves memory but slow lookup — use hash for speed.

6.5

Multi-Level (Hierarchical) Paging — Detail

▼
Concept

Page table itself is paged. Level-1 table entries point to Level-2 tables. Level-2 entries point to actual frames. PTBR (Page Table Base Register) points to Level-1 table. Only parts of the page table actually in use need to be in memory.

Rule: If page table size > desired size → add one more level. The more levels, the smaller each individual table — but more memory accesses per translation needed.
Two-level example (32-bit, 4KB pages): Logical address = 10 bits (Level-1 index) + 10 bits (Level-2 index) + 12 bits (offset).
6.6

Hashed Page Tables

▼
Concept

Virtual page number hashed → index into hash table. Each slot = linked list (chain) of entries to handle collisions. Each entry contains: (virtual page number, frame number, pointer to next). Common for 64-bit address spaces.

The virtual page number (all bits except offset bits) is used as the hash key. Linked lists handle hash collisions.
6.7

Inverted Page Table

▼
Concept

Traditional: one entry per LOGICAL page per process. Inverted: ONE entry per PHYSICAL FRAME for the entire system. Each entry = (process ID, logical page number). Only one table regardless of how many processes exist.

Advantage
  • Only one table for entire system — huge memory savings
  • Table size = number of physical frames (fixed)
Disadvantage
  • Slow lookup — must search for (pid, page-number)
  • Solution: use hash table with key = (pid, page-number)
6.8

Segmentation — Core Concept

▼
Definition

Segmentation = memory management scheme where a program is divided into variable-size logical segments based on its structure. Unlike paging (physical division), segmentation matches the programmer's logical view of a program.

Typical Segments
  • Code segment (executable instructions)
  • Data segment (global variables)
  • Stack segment (local variables, function calls)
  • Heap segment (dynamic memory allocation)
  • Shared library segments
Segment Table

OS maintains per-process segment table. Each entry has:
• Base — starting physical address
• Limit — size of the segment

Key Registers

STBR (Segment Table Base Register) — physical address of segment table in memory.
STLR (Segment Table Length Register) — number of segments. Segment s is valid only if s < STLR.

6.9

Segmentation — Address Translation

▼
Logical Address Format

<segment-number, offset>

  • 1
    CPU generates (segment-number, offset)
  • 2
    Look up segment table with segment-number → get Base and Limit
  • 3
    Validity check: Is offset < Limit? YES → valid
  • 4
    Physical Address = Base + Offset ✓
  • 5
    If offset ≥ Limit → TRAP → segmentation fault → OS terminates process
Segmentation FormulaPhysical Address = Base[segment#] + Offset MUST CHECK: Offset < Limit[segment#] (if not → segmentation fault!)
6.10

▼

6.11

Combined Segmentation and Paging

▼
Concept

Combines both: each segment is divided into pages. Gets logical view (segmentation) + non-contiguous physical storage (paging). Virtual address = segment# | page# | offset.

  • 1
    Segment number → Segment Table → starting address of that segment's page table
  • 2
    Page number → Page Table → Frame number
  • 3
    Physical Address = Frame Number + Offset

Unit 7 — Virtual Memory

Concept · Demand Paging · Page Fault · FIFO · Optimal · LRU · Belady's Anomaly · Thrashing · Paging Advantages

7.1

Virtual Memory — Concept and Purpose

▼
Definition

Virtual Memory = creates the illusion that each process has a very large private address space, even though physical RAM is limited. Uses secondary storage (disk/SSD) as an extension of RAM. Only the currently needed pages are kept in RAM.

The word "virtual" = appearing to exist but not really there. A process with 4GB virtual space doesn't need 4GB of actual RAM — it just behaves as if it has it.
Purpose 1 — Memory Extension

Programs larger than physical RAM can run. Only needed parts in RAM. Rest stays on disk until accessed.

Purpose 2 — Memory Protection

Each process gets its own virtual address space. One process cannot access another's memory — isolation guaranteed.

Limitations of Virtual Memory

  • ✗
    CPU overhead — significant overhead for address translation and page management
  • ✗
    I/O overhead — disk I/O for page swapping is millions of times slower than RAM
  • ✗
    Thrashing — system may spend more time swapping than executing if working set doesn't fit in RAM
  • ✗
    Fragmentation — still causes internal fragmentation in paging-based virtual memory
7.2

Demand Paging

▼
Definition

Demand Paging = pages are loaded into RAM ONLY when they are actually accessed (on demand). Not all pages are loaded when the program starts. "Lazy loading" strategy — load only what's needed, when it's needed.

Pure Demand Paging

NOTHING is loaded at process startup. The very first instruction causes a page fault. Very lazy — maximum laziness. Can cause many page faults at startup but allows fastest process launch time.

Demand paging allows programs much larger than RAM to run, and speeds up process startup (don't wait for everything to load).
7.3

Page Fault — Definition and Handling

▼
Definition

Page Fault = CPU tries to access a page that is NOT currently in RAM. The page table entry has its "present bit" = 0. CPU generates a trap interrupt to the OS.

Page Fault Handling Steps

  • 1
    CPU accesses virtual address. Page table shows page not present → PAGE FAULT interrupt
  • 2
    OS saves current process state (registers, etc.)
  • 3
    OS checks if the access is valid (within process's address space). Invalid → terminate process.
  • 4
    OS finds a free physical frame in RAM. If none → run page replacement algorithm to evict a page.
  • 5
    OS reads the required page from disk into the free frame
  • 6
    OS updates page table — sets frame number and present bit = 1
  • 7
    OS restores saved process state and restarts the faulting instruction
Page fault handling involves disk I/O (~10ms) vs RAM access (~100ns) — about 100,000x slower. Minimizing page faults is critical for system performance.
7.4

Page Replacement Algorithms — Overview

▼

When a page fault occurs and RAM is full, the OS must choose a victim page to evict. Goal: minimize future page faults.

FIFO — First In, First Out

Rule

The page that has been in memory the LONGEST (was loaded earliest) is replaced first. Simple queue — new pages at front, evictions from back.

Advantage
  • Very simple to implement — just a queue
  • Minimal overhead
Disadvantage
  • Belady's Anomaly: more frames can cause MORE page faults
  • Worst performance of the three algorithms
  • Old page ≠ useless page (may still be needed)

Optimal Page Replacement (OPT / MIN)

Rule

Replace the page that will NOT be used for the LONGEST time in the FUTURE. Gives minimum possible page fault rate.

Advantage
  • Gives absolute minimum page faults
  • Perfect benchmark to compare other algorithms against
Disadvantage
  • Impossible to implement in real systems — requires knowing the future sequence of page accesses
  • Only used as theoretical reference

LRU — Least Recently Used

Rule

Replace the page that has NOT been used for the LONGEST time in the PAST. Uses recent past as a predictor of near future (principle of locality of reference).

Advantages
  • Best practical algorithm — close to Optimal
  • No Belady's Anomaly (stack algorithm)
  • Solid theoretical basis (locality principle)
Disadvantages
  • More complex than FIFO
  • Requires tracking when each page was last used
  • Slight overhead
AlgorithmReplace WhichImplementableBelady'sPerformance
FIFOOldest page in memoryYesYes ⚠️Worst
OptimalFurthest future useNoNoBest (theoretical)
LRULeast recently usedYesNoNear-optimal ✓
Page Faults: FIFO ≥ LRU ≥ Optimal. LRU = best practical choice. Optimal = benchmark only.
7.5

Belady's Anomaly

▼
Definition

Belady's Anomaly = the observation that for the FIFO algorithm, increasing the number of available page frames can sometimes INCREASE the number of page faults. Counter-intuitive — normally more frames = fewer faults.

Belady's Anomaly ONLY occurs in FIFO. LRU and Optimal are "stack algorithms" — mathematically proven immune to Belady's Anomaly. Stack algorithms guarantee: more frames → same or fewer faults.
Classic example: Reference string 1,2,3,4,1,2,5,1,2,3,4,5 — FIFO with 3 frames = 9 faults, FIFO with 4 frames = 10 faults. This is Belady's Anomaly in action.
7.6

Thrashing

▼
Definition

Thrashing = a process does not have enough frames for its working set, causing continuous page faults. The OS spends more time swapping pages in/out than executing programs. CPU utilization drops to near zero. System becomes completely unresponsive.

Symptoms
  • Very high page fault rate
  • Near-zero CPU utilization
  • Disk at 100% constantly
  • System feels frozen
Solutions
  • Reduce number of running processes
  • Increase physical RAM
  • Use working set model
  • Page fault frequency control algorithm
7.7

Advantages and Disadvantages of Paging (from PPT)

▼
Advantages
  • Reduces external fragmentation completely
  • Simple to implement
  • Efficient memory management technique
  • Due to equal size pages/frames, swapping is easy
  • Supports virtual memory efficiently
Disadvantages
  • Still suffers from internal fragmentation
  • Page table requires extra memory (not good for small RAM systems)
  • Two memory accesses needed per data access (solved by TLB)
  • Page fault handling is expensive (involves disk I/O)

Device Management in an OS

8.2

Why Have Disks?

▼
Limitations of RAM
  • Small capacity (GB range)
  • Volatile — data lost when power off
  • Expensive per GB
Why Disks Are Needed
  • Large capacity (TB range)
  • Non-volatile — data persists without power
  • Cheap per GB
  • Swap space for virtual memory extension
  • File storage — OS, programs, user data

8.4

Physical Disk Structure

▼
Platter

Circular magnetic disk. Multiple platters stacked on a central spindle. Data stored magnetically on both surfaces of each platter.

Track

Concentric circular paths on each platter surface. Numbered from outside (0) inward. Each surface has thousands of tracks.

Sector

Smallest storage unit on a disk. A track divided into pie-slice sectors. Traditional = 512 bytes. Modern = 4KB (Advanced Format).

Cylinder

All tracks at the SAME RADIUS across ALL platters. All R/W heads are aligned — can access an entire cylinder without moving.

Read/Write Head

One per platter surface. Mounted on arm that moves radially to position over correct cylinder. All heads move together simultaneously.

OS View

OS simplifies disk as a large 1D array of logical blocks (0 to N-1). OS maps block number → (cylinder, track, sector). Physical complexity is hidden.

8.5

Disk Management Operations

▼
Common Operations
  • Partition a Drive — divide physical disk into logical sections
  • Format a Drive — prepare for file storage, create file system
  • Change Drive Letter — rename disk (Windows: C:, D:, etc.)
  • Shrink a Partition — reduce partition size to free space
  • Delete a Partition — remove a partition entirely
  • Change File System — NTFS, FAT32, ext4, etc.
Disk Bandwidth

Total bytes transferred ÷ total time from first request to last transfer completion. Measures overall disk I/O performance efficiency.

8.6

Disk Access Time — Three Components

▼
  • 1
    Seek Time (5–10ms) — DOMINANT COMPONENT
    Time for R/W head arm to physically move to the correct cylinder. This is the primary target of disk scheduling — minimizing seek distance minimizes seek time.
  • 2
    Rotational Latency (2–4ms)
    Time waiting for the disk to rotate until the correct sector is under the R/W head. Average = half a full rotation = (60 / RPM) / 2 seconds.
  • 3
    Transfer Time (<1ms) — SMALLEST
    Actual time to read or write the data from/to the disk surface. Depends on data size and rotation speed.
Disk Access TimeTotal Access Time = Seek Time + Rotational Latency + Transfer Time Average Rotational Latency = (1/2) × (60 seconds / RPM)
8.7

Disk Scheduling — Purpose and Goals

▼
Definition

When multiple processes request disk I/O simultaneously, the OS decides the ORDER in which to service these requests. Goal: minimize total seek time and maximize throughput while ensuring fairness.

Primary Goal

Minimize total seek time (= total head movement in cylinders). Since seek time ≈ seek distance, minimize total distance traveled by read/write head.

Secondary Goal

Fairness — prevent starvation. Ensure every request eventually gets served, even if not immediately.

Standard exam problem: Queue = {98,183,37,122,14,124,65,67}, Head at 53, Range 0–199. Every student must know this problem with all 5 algorithms.
8.8

FCFS — First Come, First Served

▼
Rule

Service requests EXACTLY in the order they arrive. No optimization whatsoever. First request = first served.

Advantages
  • Completely fair — arrival order respected
  • No starvation possible
  • Simple to implement
Disadvantages
  • Very high seek distance — head jumps randomly
  • Not efficient for moderate to heavy loads
Example result: Head at 53. Path: 53→98→183→37→122→14→124→65→67. Total = 640 cylinders.
8.9

SSTF — Shortest Seek Time First

▼
Rule

At each step, service the request with the SMALLEST distance from the current head position. Always move to the nearest unserved cylinder. Greedy algorithm.

Advantages
  • Much better than FCFS (≈236 vs 640 cylinders)
  • Minimizes individual seek times
  • Good for light to moderate loads
Disadvantages
  • STARVATION POSSIBLE — if always nearby requests exist, distant cylinders may NEVER be served
  • Similar to SJF — not globally optimal
Example result: Path: 53→65→67→37→14→98→122→124→183. Total ≈ 236 cylinders.
8.10

SCAN — Elevator Algorithm

▼
Rule

Head moves in one direction, servicing ALL requests along the way until it reaches the END OF THE DISK. Then it reverses and services in the other direction. Like an elevator going up to the top floor, then coming back down.

Advantages
  • No starvation — every request eventually served
  • Good performance under heavy load
  • Fair to most cylinder positions
Disadvantages
  • Goes to physical disk end even if last request is not there
  • Cylinders just behind the head must wait for a full sweep
Example: Head at 53 moving right → 65→67→98→122→124→183→199(end) then left → 37→14. Total ≈ 330 cylinders.
8.11

C-SCAN — Circular SCAN

▼
Rule

Head moves in ONE direction only, servicing requests. When it reaches the physical END of the disk (cylinder 199), it JUMPS back to the other end (cylinder 0) WITHOUT servicing on the way back. Then starts servicing again in the same direction. Treats disk as a circular list.

Provides MORE UNIFORM waiting times than SCAN — cylinders at both ends of the disk wait approximately the same amount of time.
Example: 53→65→67→98→122→124→183→199(end) JUMP to 0 → 14→37. Total ≈ 382 cylinders (more than SCAN due to full-width jump).
8.12

C-LOOK — Circular LOOK

▼
Rule

Like C-SCAN, but the head only goes as far as the LAST REQUEST in the current direction (NOT to the physical disk end). Then it jumps directly to the FIRST REQUEST (not cylinder 0) in the queue. No unnecessary travel to unused disk ends.

C-LOOK is an optimization of C-SCAN. Generally the best practical algorithm. Recommended as default (along with SSTF) for most systems.
C-LOOK vs C-SCAN
  • C-LOOK stops at LAST REQUEST (e.g., 183)
  • C-SCAN goes to physical disk END (e.g., 199)
  • C-LOOK jumps to FIRST REQUEST (e.g., 14)
  • C-SCAN jumps to cylinder 0
  • C-LOOK is more efficient — less wasted travel
SCAN vs C-SCAN/C-LOOK
  • SCAN goes BOTH directions
  • C-SCAN and C-LOOK go ONE direction only
  • SCAN returns with service
  • C-SCAN/C-LOOK jump back without service
  • C-SCAN/C-LOOK give more uniform wait times
Example: 53→65→67→98→122→124→183(last) JUMP to 14 → 37. Total ≈ 322 cylinders.
