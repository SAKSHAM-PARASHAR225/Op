1.1

What is Deadlock?

Imagine two cars on a one-lane bridge — one coming from the left, one from the right. Both stop and wait for the other to move back. Neither moves. This is a deadlock — a permanent standstill where no progress is possible.

Definition

Deadlock is a situation where two or more processes are permanently blocked because each process is waiting for a resource that is currently held by another waiting process. No process can proceed, and no process can release its resources.

Diagram 1 — Deadlock: Two Processes Waiting for Each Other

P1 holds Resource R1 and is waiting for R2.
P2 holds Resource R2 and is waiting for R1.
Neither will ever get what it needs. The system is stuck forever.

Deadlock Circular Wait Blocked Process Permanent Block
1.2

4 Necessary Conditions for Deadlock (Coffman Conditions)

Deadlock can only happen if ALL four conditions are true at the same time. If even one condition is false, deadlock cannot occur. These were identified by Coffman in 1971.

① Mutual Exclusion

At least one resource can be used by only one process at a time. It is non-shareable. Example: A printer can only print one job at a time.

② Hold and Wait

A process is holding at least one resource while simultaneously waiting to acquire more resources that are currently held by other processes.

③ No Preemption

Resources cannot be forcefully taken away from a process. A process only releases a resource voluntarily after it has finished using it.

④ Circular Wait

A circular chain of processes exists where each process waits for a resource held by the next process in the chain: P1→waits for→P2→waits for→P3→...→waits for→P1.

Diagram 2 — Circular Wait (The Root Cause of Deadlock)
All four conditions must be present simultaneously for deadlock. Eliminate even ONE condition and deadlock cannot happen. Deadlock prevention works by breaking one of these conditions.
Formula — Minimum Resources to Avoid Deadlock

If there are n processes and each process Pi needs a maximum of xi units of a resource:

Maximum units that ENSURE deadlock = (x₁ + x₂ + ... + xₙ) - n = Σxi - n Minimum units that ENSURE no deadlock = (Σxi - n) + 1

Why? In the worst case, each process holds (xi - 1) units and waits for 1 more. One extra unit breaks the cycle — one process finishes and releases all its resources.

Example: P1 needs max 3, P2 needs max 4, P3 needs max 5, P4 needs max 2 (n = 4) Σxi = 3 + 4 + 5 + 2 = 14 Max ensuring DEADLOCK = 14 - 4 = 10 units Min ensuring NO deadlock = 10 + 1 = 11 units ✓
Answer: With 10 units → deadlock possible. With 11 units → deadlock IMPOSSIBLE.
Remember "M-H-N-C" for the four conditions: Mutual exclusion, Hold & wait, No preemption, Circular wait.
⭐ Likely 2-Mark Question
Q: State and explain the four necessary conditions for deadlock.
Ans: (1) Mutual Exclusion — resource used by one process at a time. (2) Hold and Wait — process holds a resource while waiting for more. (3) No Preemption — resource cannot be taken forcefully. (4) Circular Wait — circular chain of waiting processes. All four must hold simultaneously.
1.3

Resource Allocation Graph (RAG)

A Resource Allocation Graph is a diagram that shows which resources are assigned to which processes, and which processes are waiting for which resources. By looking at this graph, we can check whether a deadlock has occurred.

RAG Components
  • Circles (○) = Processes (P1, P2, P3...)
  • Squares (□) = Resource types (R1, R2, R3...)
  • Dots inside squares = Individual instances of the resource
  • Arrow P → R (Request edge) = Process is WAITING for this resource
  • Arrow R → P (Assignment edge) = Resource is GIVEN to this process
Diagram 3 — RAG: With Deadlock (left) vs Without Deadlock (right)
Rules for Deadlock Detection Using RAG
  • No cycle in graph → Definitely NO deadlock
  • Cycle + only 1 instance per resource → Definitely DEADLOCK
  • Cycle + multiple instances per resource → Possible deadlock (need further checking)
Think of RAG like a map of traffic. If there's a circular traffic jam where no car can move, that is a deadlock. A one-way road system (no cycles) = no jam = no deadlock!
⭐ Likely 2-Mark Question
Q: What is a Resource Allocation Graph? Explain request and assignment edges.
Ans: RAG is a directed graph showing resource allocation state. Nodes are processes (circles) and resources (squares). A request edge (P→R) means process P is waiting for resource R. An assignment edge (R→P) means resource R is currently given to process P. If the graph has a cycle with single instances, deadlock exists.
1.4

Deadlock Handling — Four Strategies

An operating system has four ways to deal with deadlock. Think of deadlock like a road blockage — you can prevent it by better road design, avoid it by traffic control, fix it after it happens, or just ignore it!

Diagram 4 — Four Deadlock Handling Strategies

Strategy 1 — Deadlock Prevention

Design the system so that at least one of the four conditions can never occur. There are four approaches, one for each condition:

  • 1
    Break Mutual Exclusion: Make all resources shareable. In practice, this is impossible for most resources (a printer cannot print two jobs at once). Not a practical solution.
  • 2
    Break Hold and Wait: A process must request ALL resources it will ever need before it starts execution. If it cannot get all of them, it must wait and hold nothing. Drawback: Wasteful — all resources are tied up from the start even if not needed yet.
  • 3
    Break No Preemption: Allow the OS to forcefully take resources away from a waiting process. Only practical for resources whose state can be saved (like CPU registers, memory). Cannot be done for printers or tape drives.
  • 4
    Break Circular Wait (Most Practical!): Assign a number to every resource type. A process can only request resources in increasing order of numbers. This makes circular chains impossible. May cause starvation but NEVER deadlock.

Strategy 2 — Deadlock Avoidance

The OS checks before every resource allocation: "If I give this resource now, will the system remain in a safe state?" If yes, allocate. If no, make the process wait.

Safe State vs Unsafe State

Safe State: There exists at least one order (safe sequence) in which all processes can complete execution without causing deadlock. Even if we have no deadlock right now, if the state is unsafe, deadlock may occur later.

Safe → No deadlock guaranteed  |  Unsafe → Deadlock is possible (not certain)

Strategy 3 — Detection and Recovery

Allow deadlocks to happen. Periodically run a detection algorithm (check RAG for cycles, or run the safety algorithm). Once detected, recover by:

Recovery via Resources
  • Preempt a resource — take it from one process and give it to another
  • Rollback — restore a process to a previously saved checkpoint
Recovery via Processes
  • Kill one process — choose the process that has done the least work
  • Kill all deadlocked processes — drastic but complete

Strategy 4 — Deadlock Ignorance (Ostrich Algorithm)

Ostrich Algorithm

An ostrich buries its head in the sand and ignores danger. Similarly, the OS ignores the possibility of deadlock entirely. This sounds absurd, but Windows and Linux both use this approach. The reasoning: deadlocks are rare in practice. The overhead of prevention/detection is higher than the cost of occasionally rebooting when a deadlock happens.

Remember as P-A-D-I: Prevention, Avoidance, Detection, Ignorance. Four ways to handle deadlock, from most proactive to most passive.

1.5

Banker's Algorithm — Deadlock Avoidance

Think of the OS as a bank manager. A bank will only give a loan if it is confident it can recover the money later. Similarly, the OS only allocates resources if it is confident the system will stay in a safe state — that is, all processes can eventually complete.

Why "Banker's" Algorithm?

Banks never lend so much money that they cannot honor all customer withdrawals. They always keep enough in reserve. The OS does the same: it never allocates resources in a way that could lead to a deadlock.

Four Data Structures Used

Available [1 × m array]

How many free instances of each resource are currently available. Available[j] = free instances of resource type j.

Max [n × m matrix]

The maximum number of instances of each resource that each process may ever request. Declared when the process starts.

Allocation [n × m matrix]

How many instances of each resource are currently allocated to each process right now.

Need [n × m matrix] ← KEY!

Need = Max − Allocation. How many more resources each process might still request before it finishes.

Diagram 5 — Banker's Algorithm: Request Handling Flow

Safety Algorithm (Finding the Safe Sequence)

Safety Algorithm Steps
  1. Set Work = Available (copy of available resources)
  2. Set Finish[i] = false for all processes initially
  3. Find a process Pi where: Finish[i] = false AND Need[i] ≤ Work
  4. If found: Work = Work + Allocation[i] (Pi finishes and releases resources), set Finish[i] = true, go back to step 3
  5. If all Finish[i] = true → SAFE STATE ✓
  6. Otherwise → UNSAFE STATE ✗

📌 Unit 2 — Memory Management

L-2.1.1 Memory Management  |  L-2.1.2 Allocation Techniques

2.1

Address Binding — Mapping Logical to Physical Addresses

When you write a program, you use variable names and function names. The compiler converts these to addresses. Address binding is the process of mapping those program addresses to actual physical locations in RAM. This can happen at three different stages.

Definition

Address Binding is the process of mapping a program's logical (virtual) addresses to physical (main memory) addresses. Every instruction and data item in a program needs to know its exact physical location before it can be executed.

① Compile-Time Binding

The compiler produces absolute addresses. The program must always be loaded at the same memory location. Called Static Binding. If the location changes, the program must be recompiled.

② Load-Time Binding

The compiler produces relocatable code (addresses relative to 0). The loader converts them to absolute addresses when the program is loaded into memory. If the program is moved, it must be reloaded.

③ Execution-Time Binding

Address translation happens at runtime by the CPU hardware (MMU). The process can be moved in memory during execution. Called Dynamic Binding. Used by most modern operating systems.

Diagram 6 — Address Binding: From Source Code to Physical Memory
Dynamic Loading

A routine (function) is loaded into memory only when it is called. If it is never called, it is never loaded. Better memory use. No special OS support needed — done by program design.

Dynamic Linking

Linking is postponed until execution. A small stub code finds the actual library at runtime. Shared libraries (.dll in Windows, .so in Linux) use this. OS support required.

Base and Limit Registers — Hardware Protection

Base Register: Holds the starting physical address of the current process.
Limit Register: Holds the size (length) of the process.
The CPU checks every memory access: Base ≤ Address ≤ Base + Limit. If the address falls outside this range, a trap (error) is generated, protecting other processes from being corrupted.

Address Space Calculations
Physical address space = 64 KB = 2^16 bytes, Word size = 8 B = 2^3 bytes Physical address space (in words) = 2^16 / 2^3 = 2^13 words Physical address size = log₂(2^13) = 13 bits Logical address space = 128 MB = 2^27 bytes, Word size = 4 B = 2^2 bytes Logical address space (in words) = 2^27 / 2^2 = 2^25 words Logical address size = log₂(2^25) = 25 bits
General Rule: Address size (bits) = log₂(address_space_in_words)
Dynamic Loading = "load when needed" (lazy loading). Dynamic Linking = "link when needed" (lazy linking). Both save memory and time!
Logical Address Physical Address MMU Base Register Limit Register Dynamic Loading Dynamic Linking
2.2

Memory Allocation Techniques

When multiple programs run simultaneously, the OS needs a strategy to give each program its portion of RAM. This is called memory allocation. Poor allocation leads to fragmentation — wasted memory that cannot be used.

Swapping

Swapping

When RAM is full and a new process needs to run, the OS temporarily moves a process from RAM to the hard disk (swap out). Later, it brings it back (swap in). This allows more processes to run than the RAM can hold at one time. It slows the system but increases multitasking capability.

Contiguous Memory Allocation

Each process is stored in a single contiguous (uninterrupted) block of memory.

Fixed Partition Scheme
  • Memory is divided into fixed-size partitions at boot time
  • Each process is placed in one partition
  • Simple and fast
  • Problem: Internal Fragmentation — if a 5KB process is placed in an 8KB partition, 3KB is wasted inside
Dynamic (Variable) Partition Scheme
  • A partition is created exactly equal to the process size
  • No internal fragmentation
  • Problem: External Fragmentation — after many allocate/deallocate cycles, free memory is scattered in small non-contiguous pieces
Diagram 7 — Internal Fragmentation vs External Fragmentation

Dynamic Storage Allocation — First Fit, Best Fit, Worst Fit

When using dynamic partitioning, which free hole (gap) should be used for a new process? Three algorithms answer this:

① First Fit

Scan memory from the beginning and pick the first hole that is large enough. Fast but may fragment the beginning of memory over time.

② Best Fit

Search the entire list and pick the smallest hole that fits. Minimizes wasted space per allocation but leaves tiny unusable fragments. Slow.

③ Worst Fit

Search the entire list and pick the largest available hole. Leaves the biggest leftover, but is generally the least efficient overall.

Diagram 8 — Memory Allocation: First Fit vs Best Fit vs Worst Fit

Compaction

Compaction

Compaction rearranges all currently running processes in memory so that all free spaces are merged into one large contiguous block. This fixes external fragmentation. However, it is expensive — every process must be moved and its addresses updated. Requires execution-time address binding.

Diagram 9 — Compaction: Before and After
Fixed partition → Internal fragmentation (waste INSIDE). Dynamic partition → External fragmentation (waste OUTSIDE). Compaction fixes external; Paging fixes both!
Swapping Internal Fragmentation External Fragmentation First Fit Best Fit Worst Fit Compaction
⭐ Likely 2-Mark Question
Q: Differentiate between internal and external fragmentation.
Ans: Internal fragmentation occurs in fixed-partition schemes where the allocated partition is larger than the process — the unused space inside the partition is wasted. External fragmentation occurs in dynamic-partition schemes where free memory exists but is scattered in non-contiguous blocks too small to satisfy a request. Solution: paging for internal; compaction for external.

📌 Unit 3 — Paging & Segmentation

L-2.1.4 Paging, Shared Pages, Multi-Level Paging, Segmentation

3.1

Paging — Non-Contiguous Memory Management

Paging solves the problem of contiguous allocation. Think of it like a book. The book (program) is divided into pages of equal size. These pages can be placed on any shelf (physical memory frame) — they do not need to be side by side. A page table acts as the index, telling you which shelf each page is on.

Definition

Paging divides physical memory into fixed-size blocks called frames and logical memory into same-size blocks called pages. A page table maps each page number to a frame number. Any free frame can hold any page — no need for contiguous allocation. This eliminates external fragmentation.

Diagram 10 — Paging: Address Translation from Logical to Physical
Key Formulas

Page Number (p) = Logical Address ÷ Page Size

Offset (d) = Logical Address mod Page Size

Physical Address = Frame Number × Page Size + Offset

If logical address space = 2^m and page size = 2^n: Page number uses (m−n) bits, Offset uses n bits

Solved Numerical — Find Logical Address Bits
Given: 4K pages of size 512 bytes each Number of pages = 4K = 4 × 1024 = 4096 = 2^12 → Page number needs 12 bits Page size = 512 bytes = 2^9 → Offset needs 9 bits Total logical address size = 12 + 9 = 21 bits ✓
Answer: 21 bits are needed for the logical address.

Translation Lookaside Buffer (TLB)

Why TLB?

The page table is stored in main memory. So every memory access actually requires TWO memory accesses: one to look up the page table, and one to get the actual data. This doubles the access time!

The TLB (Translation Lookaside Buffer) is a small, fast hardware cache that stores recently used page table entries. If the page entry is in TLB (TLB hit), only ONE memory access is needed. If not (TLB miss), two accesses are needed.

Effective Access Time = (Hit Rate × Hit Time) + (Miss Rate × Miss Rate × Memory Access Time)

Shared Pages

Shared Pages

Multiple processes can share the same physical frame (same physical page). This is useful for common code like text editors or system libraries — only one copy exists in RAM but multiple processes use it. Shared pages must be read-only (write-protected) to prevent one process from corrupting another's data.

Multi-Level Page Tables

Why Multi-Level?

For a 32-bit address space with 4KB pages, the page table alone would need 4MB of memory per process! The solution is to break the page table into multiple levels. The level-1 table points to level-2 tables, which point to actual frames. Only the parts of the page table actually in use are kept in memory.

Paging = No external fragmentation! But may have internal fragmentation (last page not fully used). Page table itself takes memory — use TLB for speed!
Page Frame Page Table TLB Page Number Offset PTBR Internal Fragmentation

Segmentation — Logical Division of Memory

While paging divides memory physically into equal-size pieces, segmentation divides a program logically based on its structure — code section, data section, stack, heap, etc. Each part is a separate segment of different size. This matches how a programmer thinks about their program.

Definition

Segmentation is a memory management scheme where a program is divided into variable-size logical segments. Each segment has a name/number and a length. The OS maintains a Segment Table with the base address and limit for each segment.

📌 Unit 4 — Virtual Memory

L-2.1.5 Virtual Memory, Demand Paging, Page Replacement, Segmentation

4.1

Virtual Memory & Demand Paging

Have you ever wondered how a 10GB game runs on a computer with only 4GB of RAM? The answer is virtual memory. Not all parts of a program need to be in RAM at the same time. The OS keeps only the currently needed parts in RAM and stores the rest on the hard disk. When a needed part is not in RAM, the OS fetches it from disk.

Virtual Memory

Virtual memory creates an illusion that each process has a very large private address space, even though physical RAM is limited. It works by using secondary storage (hard disk/SSD) as an extension of RAM.

Two benefits: (1) Programs larger than RAM can run. (2) Memory protection — each process has its own virtual address space.

Diagram 13 — Virtual Memory: Program Larger than RAM
Demand Paging

In demand paging, a page is loaded into RAM only when the CPU actually tries to access it. Not all pages are loaded at the start — only on demand. This saves RAM and allows larger programs to run.

Page Fault: When the CPU tries to access a page that is NOT currently in RAM, a page fault interrupt occurs. The OS then fetches that page from disk, loads it into a free RAM frame, updates the page table, and restarts the instruction.

Pure Demand Paging: Nothing is loaded at startup. The very first instruction itself causes a page fault. Very lazy — load only when needed.

Diagram 14 — Page Fault: Step-by-Step Handling
Thrashing

When a process has too few frames, it causes page faults constantly. The OS spends more time swapping pages in and out than actually running programs. This is called thrashing. The system slows to a crawl. Solution: give processes enough frames, or reduce the number of running processes (lower degree of multiprogramming).

Virtual Memory Demand Paging Page Fault Swap In / Swap Out Thrashing Working Set
4.2

Page Replacement Algorithms

When a page fault occurs and RAM is full, the OS must choose which existing page to remove (evict) to make room for the new page. This decision is made by a page replacement algorithm. The goal is to minimize future page faults.

FIFO — First In, First Out

The page that has been in memory the longest time is replaced first. Like a queue — first in, first out. Simple to implement.

Problem: Belady's Anomaly — adding more frames can sometimes INCREASE page faults!

Optimal (OPT / MIN)

Replace the page that will not be used for the longest time in the future. This gives the minimum possible page faults.

Problem: Impossible to implement — you cannot know the future! Used only as a theoretical benchmark.

LRU — Least Recently Used

Replace the page that has not been used for the longest time in the past. Uses past behavior to predict future behavior (locality of reference).

Best practical algorithm! Performs close to Optimal. No Belady's Anomaly.

Diagram 15 — Page Replacement: FIFO vs LRU vs Optimal (3 Frames, Ref: 1,2,3,1,4,1,2,3,4,1)
Belady's Anomaly (FIFO only)

Normally we expect that giving a process more RAM frames should reduce page faults. With FIFO, this is sometimes NOT true — adding more frames can increase page faults. This strange behavior is called Belady's Anomaly. It is unique to FIFO. LRU and Optimal algorithms do NOT suffer from this.

📌 Unit 5 — Disk Management

L-2.2.1 Device Management, Disk Structure, Disk Scheduling Algorithms

5.1

Physical Disk Structure & Disk Access Time

A hard disk (HDD) is like a stack of circular gramophone records. Data is stored magnetically on these spinning platters, and a read/write head moves back and forth to read or write data.

Diagram 16 — Physical Hard Disk Structure
Platter

A circular magnetic disk. A hard drive has multiple platters stacked on a spindle. Data is stored on both sides of each platter.

Track & Sector

Track: Concentric circular ring on a platter. Sector: Smallest unit of storage — a pie-slice of a track. Usually 512 bytes or 4KB.

Cylinder

All tracks with the same radius across ALL platters form a cylinder. The read/write head can access an entire cylinder without moving.

Disk Access Time = Seek Time + Rotational Latency + Transfer Time

Seek Time (~5-10ms): Time for the R/W head arm to move to the correct cylinder (track). This is the largest component and the main target for optimization.

Rotational Latency (~2-4ms): Time for the disk to rotate until the correct sector is under the head.

Transfer Time (<1ms): Actual time to read or write the data from/to the sector.

Access Time = S + R + T. Seek is the slowest. Disk scheduling tries to minimize total seek time across all requests!
5.2

Disk Scheduling Algorithms

Many processes need disk access at the same time. The OS must decide the order in which to service these requests to minimize the total seek time (head movement). This is disk scheduling.

Standard Exam Problem

Request queue (cylinders): 98, 183, 37, 122, 14, 124, 65, 67
Initial head position: 53 | Disk range: 0 to 199

Diagram 17 — Disk Scheduling: Head Movement for All Algorithms

Algorithm Explanations

  • 1
    FCFS (First Come, First Served): Service requests exactly in the order they arrive. Fair but very inefficient — the head jumps randomly across the disk. Total: 640 cylinders. Works for light loads only.
  • 2
    SSTF (Shortest Seek Time First): Always move the head to the nearest request. Like a greedy algorithm. Much better than FCFS (236 cylinders) but can cause starvation — requests at far ends of the disk may never be served if there are always closer requests.
  • 3
    SCAN (Elevator Algorithm): The head moves in one direction serving all requests, reaches one end of the disk, then reverses and services in the other direction. Like an elevator. Fair, no starvation. Good for heavy loads.
  • 4
    C-SCAN (Circular SCAN): Same as SCAN but only services requests going in ONE direction. When it reaches the end, it jumps straight back to the beginning without servicing on the return trip. Provides more uniform waiting times for all cylinder positions.
  • 5
    C-LOOK: A smarter version of C-SCAN. Instead of going all the way to the disk's end (cylinder 199), the head only travels as far as the last request in that direction, then jumps to the first request in the queue. Avoids unnecessary travel. Generally the best practical algorithm!
